## Overview

Projects in vLLora provide logical isolation and organization for your AI workloads. Each project has its own configuration, credentials, usage limits, and trace data, enabling multi-tenant deployments and clean separation of concerns.

### Automatic Slug Generation

Project slugs are auto-generated from names:

```rust
fn slugify(name: &str) -> String {
    name.to_lowercase()
        .chars()
        .map(|c| if c.is_alphanumeric() { c } else { '-' })
        .collect::<String>()
        .chars()
        .fold(String::new(), |mut acc, c| {
            if !acc.ends_with('-') || c != '-' {
                acc.push(c);
            }
            acc
        })
        .trim_matches('-')
        .to_string()
}
```

**Examples:**
- `"My Project"` → `"my-project"`
- `"Production 2024"` → `"production-2024"`
- `"AI/ML Tasks"` → `"ai-ml-tasks"`

## Default Project

vLLora requires at least one default project:

```rust
pub struct ProjectServiceImpl {
    db_pool: DbPool,
}

impl ProjectService for ProjectServiceImpl {
    fn get_default(&self, _owner_id: Uuid) -> Result<Project, DatabaseError> {
        let mut conn = self.db_pool.get()?;
        DbProject::not_archived()
            .filter(schema::projects::is_default.eq(1))
            .first(&mut conn)
            .map_err(DatabaseError::QueryError)
            .map(|db_project| db_project.into())
    }
}
```

<Info>
When vLLora starts, it automatically seeds a default project if none exists (`gateway/src/seed.rs`).
</Info>

## Project Isolation

### Credential Isolation

Each project maintains its own provider credentials:

```rust
// Projects can have different API keys for the same provider
project_a.credentials.openai = "sk-proj-abc..."
project_b.credentials.openai = "sk-proj-xyz..."
```

This enables:
- **Cost allocation**: Separate billing per project
- **Rate limit isolation**: Different quotas per project
- **Security**: Credential compromise limited to one project

### Trace Isolation

Traces are tagged with project IDs:

```rust
// From telemetry/src/lib.rs
let project_id = attributes
    .remove("vllora.project_id")
    .and_then(|v| Some(v.as_str()?.to_owned()));
```

UI and queries automatically filter by project context.

// Token usage, costs, and request counts

## Project Management

### Listing Projects

```rust
fn list(&self, owner_id: Uuid) -> Result<Vec<Project>, DatabaseError> {
    let mut conn = self.db_pool.get()?;
    DbProject::not_archived()
        .load(&mut conn)
        .map_err(DatabaseError::QueryError)
        .map(|projects| projects.into_iter().map(Into::into).collect())
}
```

### Updating Projects

```rust
pub struct UpdateProjectDTO {
    pub name: Option<String>,
    pub description: Option<String>,
    pub settings: Option<Value>,
    pub archived: Option<bool>,
}

fn update(
    &self,
    id: Uuid,
    update: UpdateProjectDTO,
    _owner_id: Uuid,
) -> Result<Project, DatabaseError>
```

### Archiving Projects

Projects are soft-deleted via archiving:

```rust
DbProject::not_archived()
    .filter(schema::projects::archived_at.is_null())
```

Archived projects are hidden from lists but data is preserved.

## Multi-Tenancy

### User Allowlists

Restrict project access to specific users:

```rust
pub allowed_user_ids: Option<Vec<String>>
```

The gateway checks this list before allowing access:

```rust
if let Some(allowed_users) = &project.allowed_user_ids {
    if !allowed_users.contains(&user_id) {
        return Err(Error::Unauthorized);
    }
}
```

### Company Isolation

All projects belong to a company:

```rust
pub company_id: Uuid
```

This enables:
- Organization-level grouping
- Cross-project analytics
- Hierarchical access control
